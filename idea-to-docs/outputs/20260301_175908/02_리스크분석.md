# 리스크 분석 보고서
생성일: 2026-03-01

---

## 제품 개요

**제품명**: 링크쏙 (LinkSock) — AI 링크 저장함
**타겟 유저**: 스마트폰 사용이 서툰 50~70대 노인·중장년층
**기술 스택**: Next.js + Supabase
**개발 체계**: 1인 개발, 초기 무료, 수익화 미정

---

## 1. 기술적 리스크

### 1-1. Supabase RLS(Row Level Security) 미설정 보안 취약점

**내용**: Supabase는 클라이언트 측에서 `supabaseUrl`과 익명 키가 그대로 노출되는 구조다. RLS 정책을 올바르게 설정하지 않으면 모든 사용자의 저장 링크 데이터가 외부에 노출되거나 삭제될 위험이 있다. 1인 개발 환경에서는 보안 설정을 누락하기 쉬우며, 백엔드 지식이 부족할 경우 실제 서비스 단계에서 개인정보 관련 법적 분쟁으로 이어질 가능성이 존재한다.

**대응 방안**:
- Supabase 초기 설정 시 모든 테이블에 RLS를 반드시 활성화하고, 사용자별 정책(Policy)을 명시적으로 작성한다.
- 정기적으로 Supabase 대시보드의 Security Advisor 항목을 점검하여 미설정 항목을 확인한다.
- 클라이언트 환경 변수를 `.env.local`에 분리 관리하고, 퍼블릭 키와 시크릿 키를 혼동하지 않도록 코드 리뷰 체크리스트를 운용한다.

---

### 1-2. AI 자동 카테고리 분류 정확도 불안정

**내용**: AI 분류 모델(예: OpenAI API, Gemini API 등 외부 LLM 의존)의 경우, 노인 타겟 특유의 콘텐츠 유형(종교, 전통 민속, 특정 방언 제목 등)에서 분류 오류가 빈번할 수 있다. 잘못 분류된 링크는 UX 혼란을 유발하며, 노인 사용자는 이를 스스로 수정하거나 인지하기 어렵다.

**대응 방안**:
- 초기에는 5~7개의 단순한 상위 카테고리만 운용하여 분류 오류 영향 범위를 최소화한다.
- "미분류" 폴더를 기본으로 제공하여 분류 실패 시에도 링크가 유실되지 않도록 설계한다.
- 사용자 피드백 기반의 분류 수정 기능(카테고리 변경 버튼)을 단순하게 제공한다.

---

### 1-3. 외부 AI API 의존도 및 비용 불확실성

**내용**: OpenAI, Google Gemini 등 외부 AI API를 사용할 경우, API 가격 정책 변경, 서비스 중단, 응답 지연이 서비스 전체에 직결된다. 특히 무료 서비스임에도 AI 호출 비용이 사용자 증가에 따라 선형적으로 증가하여, 수익화 전 단계에서 개발자가 비용 부담을 고스란히 떠안는 구조다.

**대응 방안**:
- 링크 저장 시마다 AI 분류를 호출하지 않고, 큐(Queue) 방식으로 배치 처리하거나 비동기 처리로 API 호출 횟수를 최소화한다.
- 사용자당 일일 저장 한도를 초기부터 설정하여 비용 폭증을 방지한다.
- 비용 임계치 알림(billing alert)을 API 공급사 대시보드에 설정해 두고 월별 지출 한도를 고정한다.

---

### 1-4. URL 메타데이터 크롤링 불안정

**내용**: 유튜브, 일반 웹사이트에서 제목, 썸네일, 설명을 크롤링할 때 플랫폼별 HTML 구조 변경, CORS 정책, 클라우드플레어 봇 차단 등으로 메타데이터 수집이 실패할 수 있다. 특히 유튜브는 oEmbed API 정책 변경에 따라 예고 없이 응답 형식이 바뀌기도 한다.

**대응 방안**:
- 유튜브는 공식 oEmbed API 또는 YouTube Data API v3를 우선 사용하고, 실패 시 Open Graph 태그 파싱을 fallback으로 적용한다.
- 메타데이터 수집 실패 시 링크 자체는 반드시 저장되도록 설계하고, 썸네일은 기본 이미지를 대체 표시한다.

---

### 1-5. 확장성 병목

**내용**: Next.js 기반 서버리스 함수(Vercel 배포 가정)와 Supabase Free 플랜의 경우, 동시 접속자 급증 시 응답 지연 및 연결 제한(Connection Pool 부족)이 발생할 수 있다. Supabase Free 플랜은 7일 비활성화 시 프로젝트가 일시 정지되는 정책도 존재한다.

**대응 방안**:
- 서비스 초기에는 Supabase Pro 플랜 전환 기준(MAU 및 DB 용량)을 미리 정의하고, 사용량 대시보드를 주기적으로 모니터링한다.
- Supabase의 `pgBouncer` 커넥션 풀링을 활성화하여 동시 연결 이슈를 완화한다.
- 주 1회 이상 사용자 활동이 발생하도록 알림 기능(예: "새로 저장된 링크 확인하기")을 도입하여 프로젝트 비활성화를 방지한다.

---

## 2. 법적/규제 리스크

### 2-1. 개인정보보호법 적용

**내용**: 사용자가 저장하는 링크 목록과 열람 기록은 이용자의 관심사·행동 패턴을 드러낼 수 있어 개인정보보호법상 개인정보에 해당할 수 있다. 2024년 9월 시행된 개정 개인정보보호법에 따라 개인정보 처리방침을 명시하지 않거나 수집 동의 절차를 생략하면 과태료 부과 대상이 된다. AI 자동화 분류 기능은 "자동화된 결정"에 해당하여 정보주체의 설명 요구권, 의견 제출권 보장 의무가 발생할 수 있다.

**대응 방안**:
- 회원가입 단계에서 개인정보 수집 및 이용 동의를 명시적으로 받고, 처리방침 URL을 앱 하단에 상시 노출한다.
- 개인정보보호위원회의 "생성형 AI 개발·활용을 위한 개인정보 처리 안내서"(2025.8 발간)를 참고하여 AI 분류 기능의 투명성 기준을 충족한다.
- 사용자가 카테고리 분류 결과에 이의를 제기하고 수정할 수 있는 기능을 제공하여 자동화된 결정 대응권을 보장한다.

---

### 2-2. 유튜브 링크 저장 및 임베딩 관련 저작권·플랫폼 이용약관 리스크

**내용**: 2024년 11월 한국 저작권보호심의위원회는 유튜브 동영상을 임베디드 링크 방식으로 광고 없이 제공하는 앱의 불법성을 인정하고 구글 플레이 삭제 요청을 결의한 바 있다. 링크쏙이 단순 링크 저장에 그치지 않고 인앱 미리보기, 썸네일 임베딩, 광고 우회 재생 등의 기능을 제공하면 동일한 법적 위험에 노출된다.

**대응 방안**:
- 영상 재생은 반드시 유튜브 공식 앱 또는 기본 브라우저로 외부 이동(딥링크) 방식으로만 제공하고, 인앱 재생 기능을 구현하지 않는다.
- 썸네일은 YouTube oEmbed API 또는 YouTube Data API를 통해 공식적으로 취득한 이미지만 사용한다.
- YouTube API 서비스 이용약관(Terms of Service)을 정기적으로 검토하고, 정책 위반 가능 기능은 사전에 제거한다.

---

### 2-3. 개인정보 국외 이전 이슈

**내용**: Supabase 클라우드 서버 위치(기본: AWS us-east-1)와 외부 AI API(OpenAI 등)를 사용할 경우, 사용자의 링크 데이터가 국외 서버에 저장 또는 처리된다. 개인정보보호법은 국외 이전 시 사용자에게 고지하고 동의를 받거나 보호 수준 동등성을 확인할 것을 요구한다.

**대응 방안**:
- Supabase 프로젝트 생성 시 서울 리전(ap-northeast-2)을 선택하여 국내 데이터 저장을 원칙으로 한다.
- 개인정보 처리방침에 국외 이전 현황(수신자, 국가, 목적, 보호 수준)을 명시하고, 사용자에게 동의를 받는다.
- AI API 호출 시 개인 식별 정보가 포함되지 않도록 링크 URL과 공개된 메타데이터만 전송한다.

---

### 2-4. 오픈소스 라이선스 및 지식재산권

**내용**: Next.js(MIT), Supabase(Apache 2.0) 등은 상업적 사용이 허용되나, 사용하는 UI 컴포넌트 라이브러리, 아이콘 패키지, 폰트 등에 따라 상업적 배포 제한이 있을 수 있다. 추후 수익화 시 라이선스 위반이 발생할 수 있다.

**대응 방안**:
- 사용 중인 모든 패키지의 라이선스를 `license-checker` 같은 도구로 정기 점검한다.
- 상업적 이용 시 별도 계약이 필요한 라이선스(예: 일부 폰트, 아이콘)는 사전에 정품 라이선스를 구입하거나 대체재를 사용한다.

---

## 3. 시장/비즈니스 리스크

### 3-1. 타겟 사용자의 초기 유입 및 온보딩 난이도

**내용**: 50~70대 노인 사용자는 앱 설치, 회원가입, 링크 복사-붙여넣기 자체가 진입 장벽이다. 고령층의 디지털 정보화 수준은 일반 국민 대비 71.4%(2024년 기준)에 불과하며, 능숙하게 스마트폰을 다루는 역량 격차가 여전히 크다. 직접 광고로 노인 사용자를 유치하는 것은 비효율적이며, 앱 설치부터 첫 사용까지 이탈률이 매우 높을 가능성이 있다.

**대응 방안**:
- 자녀·지인이 노인 부모를 대신해 앱을 설치하고 초기 설정을 해주는 "대리 온보딩" 시나리오를 1차 유입 채널로 설계한다.
- 초기 유저 확보는 지역 복지관, 노인 디지털 교육 프로그램과의 협업을 통한 오프라인 채널에 집중한다.
- 앱 설치 없이 웹 브라우저에서도 핵심 기능을 사용할 수 있는 PWA(Progressive Web App) 방식을 검토하여 설치 허들을 낮춘다.

---

### 3-2. 대형 플랫폼의 유사 기능 흡수 가능성

**내용**: 카카오(카카오톡 링크 보관), 네이버(블로그·마이박스), 구글(구글 킵) 등이 이미 유사한 링크 저장 기능을 운영 중이다. 이들이 노인 친화 UI를 추가하거나 AI 분류 기능을 강화하면 링크쏙의 차별점이 희석될 위험이 있다.

**대응 방안**:
- 링크쏙의 핵심 가치를 "링크 저장"이 아닌 "노인이 스스로 콘텐츠를 관리하게 되는 경험"으로 포지셔닝하여 기능 모방으로 대체하기 어려운 UX 차별점을 구축한다.
- 실제 노인 사용자의 행동 데이터와 피드백을 빠르게 축적하여 경쟁사가 모방하기 어려운 노인 특화 카테고리 체계, 폰트 크기 설정, 음성 안내 등 세부 기능을 선점한다.

---

### 3-3. 수익화 경로 불확실성

**내용**: 초기 무료 정책은 사용자 확보에 유리하지만, 장기적인 수익 모델이 불명확한 상태다. 노인 사용자는 구독 결제나 인앱 결제에 익숙하지 않고, 광고 모델 적용 시 단순 UI 원칙과 충돌하여 사용성이 저하될 수 있다.

**대응 방안**:
- 자녀가 부모를 위해 "프리미엄 플랜"을 대신 결제하는 "가족 구독 모델"을 수익화 우선 옵션으로 검토한다.
- 광고보다는 B2B 채널(실버타운, 요양원, 복지관 단체 계약)을 통한 기관 단위 판매를 중기 수익화 전략으로 설정한다.

---

### 3-4. 재방문율 및 서비스 지속 사용 유도 어려움

**내용**: 링크를 저장한 후 다시 돌아와서 보는 습관이 없으면, 서비스 가치가 낮아지고 앱을 삭제하게 된다. 노인 사용자는 알림에 민감하거나 불편하게 느낄 수 있어 푸시 알림 설계가 까다롭다.

**대응 방안**:
- "오늘의 저장한 영상 다시 보기", "최근 추가한 링크" 등 주기적인 리마인더를 카카오톡 채널 또는 문자 메시지로 발송하는 방식을 검토한다.
- 앱 내에서 링크를 열람할 때마다 "잘 찾으셨나요?" 같은 단순 만족도 피드백 기능을 제공하여 사용자와의 지속적인 접점을 유지한다.

---

## 4. 운영 리스크

### 4-1. 1인 개발 체계의 지속성 리스크

**내용**: 개발자 본인의 건강, 생활 변화, 동기 저하가 서비스 유지보수와 직결된다. 장애 발생 시 즉각 대응이 어렵고, 보안 패치나 의존 라이브러리 업그레이드가 장기간 지연될 위험이 있다.

**대응 방안**:
- 온콜 대응 없이도 서비스가 운영 가능하도록 자동화된 모니터링(Sentry, Supabase Health Check)을 초기부터 설정한다.
- 핵심 서비스 로직에 대한 인수인계 문서를 작성하여, 유사시 외부 협력자나 후임 개발자가 빠르게 파악할 수 있도록 준비한다.
- 최소 유지 가능한 기능 범위(핵심 기능 3가지)를 명확히 정의하고, 그 외 기능 추가는 여력 있을 때만 진행한다.

---

### 4-2. 외부 서비스 의존 및 공급사 정책 변경 리스크

**내용**: Supabase, Vercel, OpenAI(또는 기타 AI API) 등 핵심 인프라가 모두 외부 플랫폼이다. 요금제 변경, 서비스 종료, 국내 서비스 철수 등이 발생하면 서비스 전체가 위협받는다.

**대응 방안**:
- 각 외부 서비스의 대체 옵션을 사전에 파악해 두고(예: Supabase 대체 → PlanetScale, Firebase; Vercel 대체 → Netlify, Railway), 마이그레이션 계획을 문서화한다.
- 가능한 한 특정 서비스에 Lock-in되지 않도록 추상화 레이어(Repository Pattern 등)를 적용하여 인프라 교체가 용이하도록 설계한다.

---

### 4-3. 초기 마케팅 및 채널 부재

**내용**: 1인 개발자로서 마케팅 채널, 예산, 네트워크가 부족하면 노인 사용자에게 도달하기 어렵다. 노인 타겟 특성상 기존 디지털 마케팅(검색 광고, SNS 광고) 효율이 낮다.

**대응 방안**:
- 자녀 세대(30~40대)를 2차 타겟으로 삼아, 부모님 선물용 앱으로 포지셔닝하는 바이럴 마케팅 전략을 수립한다.
- 지역 노인 복지관, 경로당 디지털 교육 강사와 협력하여 오프라인 채널로 초기 사용자 확보에 집중한다.

## 6. 상위 5개 핵심 리스크 대응 전략

### [R04] 개인정보보호법 위반 — 최우선 대응

**단계별 대응**:
1. 서비스 오픈 전: 개인정보 처리방침 문서 초안 작성 및 법률 전문가 1회 검토 의뢰(변호사 유료 자문 또는 개인정보보호위원회 무료 상담 서비스 활용)
2. 회원가입 화면: 수집 항목(이메일, 저장 링크 목록), 이용 목적, 보유 기간, 제3자 제공 여부(AI API 처리 포함)를 명시하고 동의 체크박스 구현
3. AI 분류 기능: "AI가 자동으로 카테고리를 분류합니다. 분류 결과를 변경할 수 있습니다"라는 안내 문구 및 수정 기능 제공
4. 분기 1회: 처리방침이 실제 운영 현황과 일치하는지 검토

**조기 경보 지표**: 사용자로부터 개인정보 관련 민원 접수, 앱스토어 리뷰에 개인정보 우려 내용 등장

---

### [R01] Supabase 보안 취약점 — 최우선 대응

**단계별 대응**:
1. 개발 착수 전: Supabase 대시보드에서 모든 테이블 RLS 활성화 확인 (기본값 비활성화 주의)
2. 사용자 테이블 및 링크 테이블 각각에 `auth.uid() = user_id` 조건의 RLS Policy 작성
3. 매월 1회 Supabase Security Advisor 점검 결과 확인 및 경고 항목 즉시 수정
4. 환경 변수(`.env.local`)를 Git 저장소에 커밋하지 않도록 `.gitignore` 설정 재확인

**조기 경보 지표**: Supabase 이상 쿼리 로그에서 특정 사용자 ID 없이 전체 데이터 조회 요청 감지, 비정상적인 DB 용량 증가

---

### [R02] AI API 비용 폭증 — 최우선 대응

**단계별 대응**:
1. 서비스 출시 전: 사용자당 예상 일일 링크 저장 횟수 × 호출당 비용을 시뮬레이션하여 월 예산 한도를 설정
2. API 공급사 대시보드에 월 예산 Hard Limit 설정 (OpenAI의 경우 Usage Limits 메뉴)
3. 링크 저장 즉시 AI 분류 대신 비동기/배치 처리로 전환하여 호출 빈도 최적화
4. 사용자당 일일 저장 한도(예: 20개)를 명시하고, 초과 시 다음날 처리되도록 안내

**조기 경보 지표**: 일일 API 비용이 설정 임계치의 80% 초과, 특정 시간대에 호출 수 급등

---

### [R05] 초기 사용자 온보딩 실패 — 최우선 대응

**단계별 대응**:
1. 출시 전: 실제 50~60대 사용자 3~5명에게 종이 프로토타입 또는 Figma 목업을 보여주는 사용성 테스트 진행
2. 앱 내 최초 실행 화면에 단계별 애니메이션 가이드("링크를 복사하세요 → 붙여넣기 하세요 → 저장 완료!") 구현
3. 자녀가 대신 설치할 수 있도록 "지인에게 설치 요청하기" 공유 기능 (카카오톡 공유) 초기부터 포함
4. 초기 1개월: 사용자 3명 이상의 실제 노인 사용자와 주 1회 전화 인터뷰를 진행하여 즉각 피드백 반영

**조기 경보 지표**: 첫 링크 저장까지 소요 시간이 5분 초과, 회원가입 완료율이 50% 미만, 첫 주 재방문율이 30% 미만

---

### [R03] 유튜브 저작권 및 플랫폼 정책 위반 — 높음 대응

**단계별 대응**:
1. 기능 설계 단계: 인앱 동영상 재생 기능 구현 금지, 모든 재생은 외부 앱(유튜브 앱 또는 기본 브라우저)으로 이동하는 딥링크만 사용
2. 유튜브 썸네일 및 제목 취득 시 YouTube Data API v3 또는 공식 oEmbed API만 사용하고, YouTube API 서비스 이용약관 준수
3. 앱스토어 제출 전: YouTube API Terms of Service 체크리스트를 직접 검토하고 위반 소지 기능을 사전에 제거
4. 저작권 관련 사용자 신고 기능 제공 및 신고 접수 시 24시간 내 검토 프로세스 수립

**조기 경보 지표**: YouTube API 할당량 초과 경고 수신, 저작권 관련 사용자 민원 접수, 앱스토어 정책 위반 경고 이메일 수신
