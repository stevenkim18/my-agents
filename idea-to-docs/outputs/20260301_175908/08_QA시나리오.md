## 1. 테스트 전략

### 1.1 테스트 범위 및 목표

**제품명**: 링크쏙 (LinkSock) — AI 링크 저장함
**플랫폼**: 모바일 웹 (Next.js 14 + Supabase)
**주요 타겟 사용자**: 50~70대 노인·중장년층

**테스트 목표**
- MVP 7개 핵심 기능의 정상 동작 검증
- 시니어 사용자 UX 기준 충족 여부 확인 (18px+ 폰트, 버튼 44×44px+)
- 보안 취약점 사전 차단 (RLS 정책 검증)
- 서비스 안정성 및 오류 복구 능력 검증
- 첫 링크 저장까지 3분 이내 완료 가능 여부 검증

**테스트 범위 내 (In-Scope)**

| 항목 | 상세 |
|------|------|
| 기능 테스트 | F01~F07 전 기능 |
| 엣지케이스 | 경계값, 빈 입력, 네트워크 오류, 중복 처리 |
| 보안 테스트 | RLS 정책, 세션 관리, 권한 검증 |
| UX 접근성 | 폰트 크기, 버튼 크기, 터치 영역 |
| 성능 테스트 | API 응답 시간, 동시 사용자 부하 |
| 회귀 테스트 | 배포 전 핵심 기능 체크리스트 |

**테스트 범위 외 (Out-of-Scope)**
- 네이티브 앱 (iOS/Android)
- IE 브라우저 지원
- 관리자 백오피스 기능

### 1.2 테스트 유형

| 테스트 유형 | 도구 | 담당 영역 |
|------------|------|----------|
| 단위 테스트 | Jest + React Testing Library | 컴포넌트, 유틸 함수, API 파싱 로직 |
| 통합 테스트 | Jest + Supabase Mock | DB 연동, RLS 정책, Auth 플로우 |
| E2E 테스트 | Playwright | 전체 사용자 시나리오 (저장~조회~삭제) |
| API 테스트 | Postman / Thunder Client | OG 파싱, AI 분류 Edge Function |
| 성능 테스트 | k6 / Artillery | 동시 사용자 부하, API 응답 시간 |
| 보안 테스트 | 수동 + OWASP 기준 | RLS 우회 시도, 인증 토큰 검증 |
| UX 접근성 검증 | Chrome DevTools + 수동 | 폰트 크기, 버튼 크기, 색상 대비 |

### 1.3 테스트 환경 및 도구

**환경 구성**

| 환경 | URL | 용도 |
|------|-----|------|
| 개발 (DEV) | localhost:3000 | 개발자 자체 테스트 |
| 스테이징 (STG) | staging.linksock.com | QA 팀 전체 테스트 |
| 운영 (PROD) | linksock.com | 최종 스모크 테스트 |

**테스트 기기 최소 요구 사항**
- Android: Galaxy A 시리즈 (화면 크기 6인치 이상) — Chrome 최신
- iOS: iPhone 12 이상 — Safari 최신
- 태블릿: 시니어 사용 빈도 고려하여 iPad 포함

**Supabase 테스트 설정**
- 테스트 전용 프로젝트 사용 (운영 DB 격리)
- RLS 정책 스테이징 환경 동일하게 적용
- Edge Function 로컬 에뮬레이터: `supabase functions serve`

## 2. 기능 테스트 시나리오

### F01: 링크 붙여넣기 저장

---

```
TC-001: 유효한 URL 클립보드 저장 정상 흐름
기능: F01 링크 붙여넣기 저장
유저 스토리: US-01

사전 조건:
- 로그인된 사용자 세션이 존재한다
- 클립보드에 유효한 URL이 복사되어 있다 (예: https://www.youtube.com/watch?v=abc123)
- 당일 저장 횟수가 0개이다

테스트 단계:
1. 앱 메인 화면에 접근한다
2. "링크 저장" 버튼을 탭한다
3. 클립보드에서 자동으로 URL이 입력되는지 확인한다
4. "저장했어요!" 피드백 메시지가 표시되는지 확인한다
5. 저장된 링크가 목록에 추가되는지 확인한다

기대 결과:
- 버튼 1회 탭으로 저장이 완료된다
- "저장했어요!" 텍스트 피드백이 화면에 표시된다
- 목록 상단에 새 링크가 나타난다
- AI 분류가 비동기로 진행되며, 완료 전까지는 "분류 중" 상태가 표시된다
우선순위: Critical
```

### F03: AI 자동 카테고리 분류

---

```
TC-008: AI 자동 카테고리 분류 정상 동작
기능: F03 AI 자동 카테고리 분류
유저 스토리: US-03

사전 조건:
- 로그인된 사용자가 있다
- 링크 저장이 완료된 상태이다
- AI API (gpt-4o-mini 또는 Claude Haiku)가 정상 동작 중이다

테스트 단계:
1. 뉴스 URL을 저장한다 (예: 정치·사회 기사)
2. 목록에서 해당 링크의 카테고리 표시를 확인한다
3. 비동기 AI 분류가 완료될 때까지 기다린다 (최대 10초)
4. 분류된 카테고리 레이블이 표시되는지 확인한다

기대 결과:
- 저장 직후: "분류 중" 상태가 표시된다
- AI 분류 완료 후: 적절한 카테고리(예: "뉴스")가 Realtime으로 업데이트된다
- 7개 카테고리 중 하나로 분류된다 (뉴스, 쇼핑, 유튜브, 레시피, 여행, 건강, 기타)
우선순위: High
```

```
TC-009: AI 분류 완료 후 사용자 수동 카테고리 변경

3. "건강" 카테고리를 선택한다

- 카테고리가 "뉴스"에서 "건강"으로 변경된다
- 목록 새로고침 없이 즉시 변경 사항이 반영된다
- 카테고리 탭 필터에서도 올바르게 이동된다
우선순위: Medium
```

### F05: 회원가입/로그인

---

```
TC-013: 이메일 회원가입 정상 흐름
기능: F05 회원가입/로그인
유저 스토리: US-05

사전 조건:
- 사용자가 앱에 처음 접근한다
- 해당 이메일로 가입된 계정이 없다
```
