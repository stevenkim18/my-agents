# MVP 스코핑 보고서
생성일: 2026-03-01

---

## 제품 개요

**제품명**: 링크쏙 (LinkSock) — AI 링크 저장함
**타겟**: 스마트폰 사용이 서툰 50~70대 노인·중장년층
**개발 체계**: 1인 개발 / Next.js + Supabase / 초기 무료

---

## 1. MVP 전략 방향

### 1.1 MVP를 통해 검증하려는 핵심 가설

| 번호 | 핵심 가설 | 검증 방법 |
|------|-----------|-----------|
| H1 | 50~70대 사용자가 링크 붙여넣기 하나로 링크를 저장하는 행동을 반복할 것이다 | 2주 후 재방문율 및 저장 반복 여부 측정 |
| H2 | AI 자동 카테고리 분류가 없으면 사용자가 저장된 링크를 다시 찾지 못한다 | 카테고리 탭 클릭률 및 링크 재열람율 측정 |
| H3 | 자녀가 부모님 스마트폰에 앱을 설치해주는 "대리 온보딩" 패턴이 주 유입 채널이 될 것이다 | 신규 가입 경로(직접/공유링크) 추적 |
| H4 | 큰 글씨·큰 썸네일 UI가 기존 앱 대비 이탈률을 낮출 것이다 | 첫 저장까지 소요 시간 및 이탈 단계 측정 |

### 1.2 타겟 Early Adopter 정의

**1차 얼리어답터: 자녀의 도움을 받는 디지털 초보 시니어**
- 60~72세, 스마트폰 보유 but 앱 설치/조작 어려움
- 카카오톡으로 자녀·지인에게 영상 링크를 받지만 저장 방법을 모름
- 유튜브 사용 빈도 주 3회 이상 (건강, 요리, 종교, 뉴스 콘텐츠 소비)
- 자녀(30~40대)가 앱 설치 및 초기 셋업을 도와주는 환경

**2차 얼리어답터: 스스로 디지털 도구를 탐색하는 액티브 시니어**
- 55~65세, 앱스토어에서 직접 앱을 검색·설치 가능
- 기존 네이버 메모나 유튜브 재생목록에 불편함을 느끼는 사용자

### 1.3 MVP 성공 기준 (정량 지표)

| 지표 | 목표 (출시 후 8주 기준) | 피봇 기준 |
|------|----------------------|-----------|
| 총 가입자 수 | 200명 이상 | 50명 미만 시 온보딩 방식 전환 검토 |
| 1주 재방문율 | 40% 이상 | 20% 미만 시 재방문 루프 재설계 |
| 2주 재방문율 | 30% 이상 | 15% 미만 시 핵심 가치 재검토 |
| 사용자당 평균 저장 링크 수 | 5개 이상 | 2개 미만 시 저장 흐름 간소화 재검토 |
| 첫 링크 저장까지 소요 시간 | 3분 이내 | 5분 초과 시 UX 개선 필수 |
| 앱스토어 평점 | 4.0 이상 | 3.5 미만 시 즉시 부정 리뷰 원인 분석 |
| 실제 50세 이상 사용자 비율 | 70% 이상 | 50% 미만 시 타겟팅 전략 재검토 |

## 2. 기능 분류 (MoSCoW)

### Must Have — MVP에 반드시 포함되어야 하는 기능

> 이 기능들이 없으면 제품이 핵심 가치를 전달하지 못하거나 서비스가 동작하지 않는다.

#### M1. 링크 붙여넣기 저장
- 클립보드의 URL을 붙여넣기 하면 즉시 저장되는 단일 인터랙션
- 유튜브 URL, 일반 웹사이트 URL 모두 지원
- 저장 완료 시 시각적 피드백 (큰 글씨 "저장했어요!" 메시지)
- **근거**: 제품의 핵심 존재 이유. 이 기능 없이 나머지 기능은 무의미

#### M2. URL 메타데이터 자동 파싱
- 유튜브: oEmbed API 또는 YouTube Data API v3로 제목·썸네일 취득
- 일반 웹사이트: Open Graph 태그 파싱으로 제목·썸네일 취득
- 파싱 실패 시 기본 이미지와 URL 텍스트로 표시 (저장 자체는 항상 성공)
- **근거**: 썸네일·제목 없이는 노인 사용자가 저장된 링크를 구별하지 못함

#### M3. AI 자동 카테고리 분류 (5~7개 고정 카테고리)
- 초기 카테고리: 건강, 요리·음식, 뉴스·시사, 종교·신앙, 여행·취미, 가족·생활, 기타
- 링크 저장 시 비동기 처리로 카테고리 자동 분류
- 분류 실패 또는 불확실 시 "기타"로 자동 배정 (저장 실패 없음)
- 사용자가 카테고리 변경 가능 (단순 드롭다운)
- **근거**: 직접 폴더 관리가 불가능한 노인 사용자에게 분류 자동화는 필수 차별점

#### M4. 저장된 링크 목록 보기 (큰 글씨·큰 썸네일 UI)
- 최소 글씨 크기 18px 이상
- 썸네일 크기 최소 120×90px 이상
- 카테고리 탭으로 필터링 가능
- 링크 터치 시 외부 앱(유튜브 앱 또는 기본 브라우저)으로 이동 (딥링크)
- **근거**: 저장한 링크를 다시 찾아서 열어볼 수 없다면 서비스 가치가 0

#### M5. 회원 가입 및 로그인 (이메일 또는 카카오)
- 이메일 + 비밀번호 로그인
- 카카오 소셜 로그인 (노인 친숙도 고려)
- Supabase Auth 활용
- **근거**: 데이터 영속성 및 디바이스 간 접근을 위한 최소 조건

#### M6. 기본 보안 설정
- Supabase RLS 전 테이블 활성화 (사용자별 데이터 격리)
- 서울 리전(ap-northeast-2) 데이터 저장
- **근거**: 리스크 분석 R01/R04 최우선 대응. 보안 없이는 서비스 출시 불가

#### M7. 링크 삭제
- 저장된 링크를 삭제하는 단순 기능 (길게 누르기 또는 삭제 버튼)
- **근거**: 잘못 저장하거나 필요 없어진 링크를 정리하지 못하면 앱이 쓰레기통이 됨

---

### Should Have — MVP 이후 빠르게 추가해야 할 기능 (2~3개월 이내)

> 없어도 서비스가 동작하지만, 이 기능들이 없으면 경쟁력 또는 재방문율이 크게 저하된다.

#### S1. 카카오톡 공유 연동 (Share Extension)
- iOS Share Sheet / Android Intent로 카카오톡에서 바로 링크쏙으로 전송
- **근거**: 시니어의 링크 수신 채널 1위가 카카오톡. 이 연동 없이는 "복사 → 앱 열기 → 붙여넣기" 3단계가 필수라 이탈 높음

#### S2. 재방문 유도 알림
- "새로 저장한 영상이 있어요" 주 1회 푸시 알림
- "오늘의 건강 영상" 등 카테고리별 리마인더 (선택 설정)
- **근거**: 리스크 분석 R14(재방문율 저조) 대응. 저장 후 재방문 없으면 서비스 가치 미검증

#### S3. 앱 내 온보딩 튜토리얼
- 최초 실행 시 3단계 슬라이드 ("링크 복사 → 붙여넣기 → 저장 완료")
- 실제 저장 시연을 포함한 인터랙티브 가이드
- **근거**: 리스크 분석 R05(온보딩 실패) 대응. 초기 이탈 방지에 직결

#### S4. 링크 검색
- 저장된 링크의 제목 기반 텍스트 검색
- 큰 검색창, 큰 글씨 결과 목록
- **근거**: 저장 링크가 20개 이상 쌓이면 카테고리 탭만으로는 찾기 어려워짐

#### S5. 개인정보 처리방침 및 동의 화면
- 회원가입 시 수집 항목, 이용 목적, 보유 기간 명시 동의
- AI 자동 분류 안내 문구 및 수정 가능성 고지
- **근거**: 리스크 분석 R04 대응. 법적 의무 사항이나 MVP 출시 직후 즉시 추가 가능

### Nice to Have — 향후 로드맵 기능 (3개월 이후)

> 차별화를 강화하거나 특정 세그먼트 수요를 충족하는 부가 기능.

#### N1. 가족 공유 기능
- 자녀가 부모님 계정에 링크를 직접 추가해주는 기능
- "가족 큐레이션" — 자녀가 엄선해서 보내주는 링크 탭
- **근거**: 비즈니스 분석상 가족 플랜 수익화 모델의 핵심 기능

#### N2. AI 한 줄 요약
- 저장된 링크의 콘텐츠를 AI가 2~3문장으로 요약
- 긴 영상/기사의 핵심만 보려는 시니어 니즈
- **근거**: API 비용 증가 리스크(R02)가 있어 수익화 이후 도입이 적절

#### N3. 음성 검색
- "건강 영상 보여줘"처럼 음성으로 저장된 링크 검색
- **근거**: 고령 사용자 타이핑 어려움 해소. 기술 구현 난이도 높아 우선순위 후순위

#### N4. 잠금 화면 위젯 / 홈 화면 위젯
- iOS/Android 위젯으로 최근 저장 링크 빠르게 접근
- **근거**: 앱 열기 단계 자체를 줄이는 UX. 개발 난이도 상

#### N5. PWA (Progressive Web App) 지원
- 앱 설치 없이 브라우저에서 바로 사용 가능
- **근거**: 리스크 분석 R05(온보딩 허들 완화). Next.js 기반이라 기술적으로 용이

#### N6. B2B 기관 대시보드
- 복지관·실버타운 담당자가 단체 계정을 관리하는 어드민 화면
- **근거**: 비즈니스 분석상 중기 수익화 채널. 개인 사용자 기반 확보 후 검토

#### N7. 소프트 페이월 (저장 30개 한도 → 유료 전환)
- 무료 사용자는 저장 30개 한도, 이후 프리미엄 업그레이드 유도
- **근거**: 비즈니스 분석 권고 수익화 모델. 사용자 1,000명 이상 확보 후 도입 권고

### Won't Have — 이번 MVP에서 명시적으로 제외

| 기능 | 제외 이유 |
|------|-----------|
| 인앱 유튜브 재생 | 저작권법 위반·앱스토어 퇴출 리스크(R03). 외부 딥링크만 허용 |
| 태그/폴더 직접 관리 | 노인 사용자 UX 원칙 위배. AI 자동 분류로 대체 |
| 소셜/커뮤니티 기능 (좋아요, 댓글) | MVP 범위 초과. 1인 개발 리소스 부족 |
| 크롬 익스텐션 | 노인 타겟 사용 패턴과 맞지 않음. 우선순위 낮음 |
| 콘텐츠 오프라인 저장 | 저작권 문제 및 개발 복잡도 증가 |
| 다국어 지원 | 초기 타겟은 한국 사용자만. 글로벌 확장은 3년+ 이후 |
| 광고 모델 | 단순 UI 원칙 위배. 시니어 UX 훼손 |

## 3. MVP 개발 우선순위 및 의존 관계

### 3.1 기능별 개발 난이도

| 기능 | 난이도 | 예상 개발 시간 | 의존 기능 |
|------|--------|--------------|-----------|
| M5. 회원가입/로그인 | 하 | 1~2일 | 없음 (Supabase Auth 기본 제공) |
| M6. 보안 설정 (RLS) | 하 | 반나절 | M5 |
| M1. 링크 붙여넣기 저장 | 하 | 1일 | M5, M6 |
| M2. URL 메타데이터 파싱 | 중 | 2~3일 | M1 |
| M3. AI 자동 카테고리 분류 | 중 | 2~3일 | M1, M2 |
| M4. 링크 목록 보기 (시니어 UI) | 중 | 3~4일 | M1, M2, M3 |
| M7. 링크 삭제 | 하 | 반나절 | M4 |

**총 Must Have 예상 개발 기간**: 약 2~3주 (1인 풀타임 기준)

## 4. MVP 개발 마일스톤

### 마일스톤 0 — 기반 셋업 (1주차)

**목표**: 개발 환경 구성 및 데이터 모델 확정

- [ ] Next.js 프로젝트 초기화 및 Supabase 프로젝트 생성 (서울 리전)
- [ ] Supabase 테이블 설계: `users`, `links`, `categories`
- [ ] RLS 정책 전 테이블 활성화 및 Policy 작성
- [ ] Supabase Auth 설정 (이메일 + 카카오 소셜 로그인)
- [ ] Vercel 배포 파이프라인 연결
- [ ] 환경 변수 `.env.local` 분리 및 `.gitignore` 설정
- [ ] AI API 비용 모니터링 알림 설정 (월 한도 지정)

**완료 기준**: 로컬 환경에서 로그인이 작동하고 Supabase에 데이터가 저장됨

---

### 마일스톤 1 — 저장 코어 (2주차)

**목표**: 링크 저장의 핵심 흐름 완성

- [ ] 링크 입력 화면 UI (큰 붙여넣기 버튼, 18px+ 폰트)
- [ ] 클립보드 자동 감지 및 URL 유효성 검사
- [ ] 유튜브 oEmbed API 연동 (제목·썸네일 파싱)
- [ ] 일반 웹사이트 Open Graph 파싱 (서버사이드 fetch)
- [ ] 파싱 실패 시 기본 이미지 fallback 처리
- [ ] 저장 완료 피드백 ("저장했어요!" 큰 글씨 메시지)
- [ ] Supabase `links` 테이블에 데이터 insert

**완료 기준**: 유튜브 URL과 일반 URL을 붙여넣기 하면 썸네일과 제목이 저장됨

---

### 마일스톤 2 — AI 분류 및 목록 UI (3~4주차)

**목표**: AI 카테고리 분류와 시니어 특화 UI 완성

- [ ] AI 카테고리 분류 API 연동 (비동기 처리)
  - 카테고리 7개: 건강, 요리·음식, 뉴스·시사, 종교·신앙, 여행·취미, 가족·생활, 기타
  - 저장 즉시 "분류 중..." 표시 → 완료 시 카테고리 뱃지 업데이트
- [ ] AI 분류 실패 시 "기타"로 자동 배정 로직
- [ ] 링크 목록 화면 (카드 레이아웃, 큰 썸네일)
- [ ] 카테고리 탭 필터 (탭 버튼 크기 최소 44px)
- [ ] 링크 터치 시 외부 앱으로 딥링크 이동
- [ ] 링크 삭제 기능 (길게 누르기 또는 삭제 버튼)
- [ ] 사용자가 카테고리 수동 변경하는 기능

**완료 기준**: 저장된 링크가 카테고리별로 분류되어 큰 UI로 표시됨

### 마일스톤 3 — 베타 검증 (5~6주차)

## 7. MVP 범위 요약 및 우선순위 근거

### 핵심 원칙: "저장 → 찾기" 2단계만 완벽하게

링크쏙 MVP의 본질은 단 하나다. **노인이 링크를 받았을 때 잃어버리지 않고, 나중에 다시 볼 수 있게 하는 것.** 이 두 단계(저장, 재열람)가 매끄럽게 작동한다면 MVP는 성공이다. 그 외 모든 기능은 이 두 단계의 경험을 강화하거나 사용자 규모를 늘리는 보조 수단이다.

### 기능 범위 판단 근거

1. **1인 개발 제약**: 6~8주 안에 안정적으로 작동하는 제품을 출시하려면 Must Have 7개 기능에만 집중해야 한다. 기능이 많을수록 버그 발생 가능성이 높고, 노인 사용자는 작은 오류에도 앱을 삭제한다.

2. **리스크 우선 대응**: 비즈니스/리스크 분석에서 최우선(R01~R05)으로 분류된 리스크를 MVP 단계에서 해소하지 않으면 서비스 지속 자체가 불가능하다. 보안(M6), 저작권 준수(딥링크 방식), 개인정보 처리방침(S5)이 Must-Should 경계에 있는 이유다.

3. **가설 검증 우선**: 카카오톡 연동(S1)처럼 임팩트가 크지만 구현에 2~3일이 필요한 기능은 "가설 H3(자녀 채널 유입)"을 검증한 후 추가하는 것이 합리적이다. 가설이 틀렸다면 이 기능의 우선순위가 달라진다.

4. **노인 UX 집중**: 모든 Must Have 기능의 디자인 기준은 "스마트폰이 서툰 65세가 혼자서 조작 가능한가"이다. 기능 수를 줄이는 대신, 남은 기능 하나하나의 UX 완성도를 높이는 데 개발 리소스를 투자한다.
