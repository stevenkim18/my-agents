# 리스크 분석 보고서
생성일: 2026-03-04

## 대상 제품: WeCal (함께 쓰는 캘린더)

---

## 1. 기술적 리스크

### 1-1. 기술 구현 난이도 및 불확실성

**실시간 동기화 구현의 복잡도**
- 1인 개발 환경에서 실시간 데이터 동기화(WebSocket, Firebase Realtime Database 또는 Firestore)를 안정적으로 구현하는 것은 상당한 난이도를 요구한다.
- 오프라인 상태에서 일정을 수정한 후 온라인 복귀 시 충돌(Conflict) 해결 로직이 필요하다. 예를 들어 두 사람이 동시에 같은 일정을 수정할 경우 어떤 버전을 기준으로 반영할 것인지 정책이 없으면 데이터 불일치가 발생한다.
- **위험 수준:** 높음

**공동 일정 수락/거절 플로우 구현**
- 초대 → 알림 발송 → 수락/거절 → 양측 캘린더 반영까지 이어지는 상태 머신이 복잡하다. 네트워크 지연, 앱 미설치, 알림 권한 미부여 등 엣지 케이스가 다수 존재한다.
- **위험 수준:** 중간

### 1-2. 외부 기술/서비스 의존성

**Google Calendar API 의존성**
- Google Calendar API는 프로젝트 단위로 일일 1,000,000 쿼리, 사용자당 분당 쿼리 제한(Rate Limit)이 존재한다.
- 사용자가 급증하면 Rate Limit 초과(HTTP 403/429) 오류가 발생할 수 있으며, 이는 핵심 기능 중단으로 이어진다.
- Google이 정책 없이 API 스펙을 변경하거나 서비스를 중단할 경우(2021년에도 API 관리 방식이 변경된 사례가 있음) 연동 기능 전체가 장애를 일으킬 수 있다.
- **위험 수준:** 중간

**Apple APNs / Firebase Cloud Messaging 의존성**
- 푸시 알림은 APNs를 통해 발송된다. Apple 개발자 계정(연 $99)이 필요하며, 인증서 만료 시 알림이 전면 중단된다.
- FCM(Firebase Cloud Messaging)을 중간 레이어로 사용할 경우, Google의 FCM 정책 변경이나 Firebase 서비스 장애 시 알림 기능이 중단될 리스크가 있다.
- **위험 수준:** 중간

**Firebase / Supabase 등 BaaS 의존성**
- 1인 개발 특성상 직접 서버를 운영하지 않고 BaaS를 사용할 가능성이 높다.
- BaaS 플랫폼의 요금 정책 변경(Firebase가 2023년 Spark 플랜 스토리지 한도를 조정한 사례 등)이나 서비스 종료 시 마이그레이션 비용이 크다.
- **위험 수준:** 중간

### 1-3. 보안 취약점

**일정 데이터 무단 접근**
- 공유 그룹 내 데이터 접근 제어가 잘못 설정되면 다른 그룹의 일정이 노출될 수 있다. 특히 커플/가족의 민감한 일정(병원 방문, 개인 약속 등)이 유출될 경우 치명적인 신뢰 손실이 발생한다.
- Firestore/Realtime DB의 보안 규칙을 잘못 작성하면 인증 없이 전체 데이터를 읽을 수 있는 취약점이 생긴다.
- **위험 수준:** 높음

**Google OAuth 토큰 관리**
- 외부 캘린더 연동 시 OAuth 2.0 액세스 토큰과 리프레시 토큰을 서버 또는 기기에 저장하게 된다. 토큰 탈취 시 사용자의 Google 캘린더 전체에 접근할 수 있다.
- **위험 수준:** 높음

### 1-4. 확장성 병목 지점

- 실시간 동기화를 위한 WebSocket 연결이나 Firestore 리스너는 동시 접속자 수 증가에 따라 비용이 선형 이상으로 증가할 수 있다.
- 초기에는 무료 티어로 운영 가능하지만, 사용자가 수천 명을 넘어서면 인프라 비용이 급격히 증가하는 구조가 될 수 있다.
- **위험 수준:** 중간 (초기 단계에서는 낮음, 성장 후 높아짐)

## 2. 법적/규제 리스크

### 2-1. 개인정보보호법 (PIPA) 준수

**민감 정보 처리**
- 일정 정보는 개인의 동선, 건강 상태, 대인 관계를 추론할 수 있는 민감 정보에 해당할 수 있다.
- 2025년 개인정보보호법 3차 개정(개인정보 전송 요구권 포함)에 따라 사용자가 자신의 데이터 이동을 요구할 경우 이에 응해야 하는 의무가 생긴다.
- 개인정보 처리방침 공시, 수집 목적 명시, 제3자 제공 동의 등 법적 의무를 이행하지 않으면 과태료(최대 3,000만 원) 또는 과징금(매출액의 3%) 처분을 받을 수 있다.
- **위험 수준:** 높음

**데이터 보관 및 삭제**
- 탈퇴한 사용자의 데이터를 즉시 삭제하지 않거나, 보관 기간을 명시하지 않으면 법적 문제가 발생한다.
- **위험 수준:** 중간

### 2-2. Apple App Store 정책 위반 가능성

**Sign in with Apple 필수 요건**
- Google, Facebook 등 제3자 소셜 로그인을 제공할 경우 반드시 "Sign in with Apple"도 함께 제공해야 한다(App Store 심사 지침 4.8조). 미준수 시 앱 심사 거절된다.
- **위험 수준:** 중간

**인앱 결제(IAP) 정책**
- 프리미엄 기능이나 구독 상품을 제공할 경우 반드시 Apple IAP를 통해야 하며, Apple이 30%(첫 해는 15%) 수수료를 부과한다. 수익화 모델 설계 시 이를 반드시 고려해야 한다.
- **위험 수준:** 중간

**앱 완성도 미달 심사 거절**
- 실시간 동기화, 알림 등 핵심 기능이 불안정하면 심사 중 반복 충돌로 리젝될 수 있다.
- **위험 수준:** 중간

### 2-3. Google API 서비스 약관 (Terms of Service)

- Google Calendar API를 사용할 경우 Google API 서비스 약관을 준수해야 한다. 사용자 데이터를 수집하거나 제3자에게 제공하는 것은 엄격히 제한된다.
- OAuth 인증 범위(Scope)를 실제 필요 이상으로 요청하면 Google의 OAuth 앱 검토 프로세스에서 승인이 거부될 수 있다. 특히 캘린더 읽기/쓰기 권한은 민감한 범위(Sensitive Scope)로 분류되어 추가 검토가 필요하다.
- **위험 수준:** 높음

### 2-4. 지식재산권 이슈

- UI/UX 디자인이 기존 경쟁 앱(TimeTree, Between 등)의 독특한 디자인 요소를 무단으로 모방할 경우 저작권 침해 주장을 받을 수 있다.
- **위험 수준:** 낮음

### 2-5. 국내외 규제 차이

- 향후 글로벌 서비스 확장 시 EU GDPR 준수가 요구된다. GDPR은 한국 개인정보보호법보다 요건이 엄격하며 위반 시 전 세계 매출의 4% 또는 2,000만 유로 중 높은 금액의 과징금이 부과된다.
- **위험 수준:** 낮음 (초기 국내 서비스 기준; 글로벌 확장 시 높아짐)

## 3. 시장/비즈니스 리스크

### 3-1. 시장 수요 불확실성

- 공유 캘린더 앱 시장은 2025년 기준 약 6,890만 달러 규모로, 연평균 11~15% 성장 중이다.
- 그러나 TimeTree(6,000만 사용자), Cozi(2,000만 이상 가족 등록)처럼 이미 강력한 플레이어가 글로벌 시장을 점유하고 있다.
- 커플/소규모 그룹에 특화된 니치 포지셔닝이 유효하지만, 실제 유료 전환 의향이 있는 사용자 규모는 불확실하다.
- **위험 수준:** 중간

### 3-2. 대형 경쟁사의 진입/복제 가능성

- Apple이 iOS 기본 캘린더 앱에 공유 기능을 강화하거나, Google이 Google Calendar에 커플/가족 특화 기능을 추가할 경우 차별점이 크게 희석된다.
- 네이버, 카카오 같은 국내 대형 플랫폼이 유사 기능을 자사 앱(카카오캘린더 등)에 통합할 경우 기존 사용자 기반을 활용해 빠르게 시장을 가져갈 수 있다.
- **위험 수준:** 높음

### 3-3. 수익화 실패 시나리오

- 수익화 모델이 미정인 상태로 출시하면 지속 가능한 비즈니스 모델 없이 운영될 위험이 있다.
- 공유 캘린더는 특성상 "무료여야 한다"는 사용자 인식이 강하여 유료 전환이 어렵다. Apple IAP 구독 수수료(15~30%)까지 고려하면 수익성이 더 낮아진다.
- 광고 기반 수익화를 선택할 경우 커플/가족 관계 정보에 기반한 타겟 광고는 개인정보 침해 우려를 유발하고 앱에 대한 부정적 인식을 줄 수 있다.
- **위험 수준:** 높음

### 3-4. 고객 이탈 요인

- 공유 캘린더 앱의 특성상 "두 명 모두" 앱을 설치하고 적극적으로 사용해야 한다. 한 명이라도 이탈하면 서비스 가치가 급감하는 네트워크 효과의 역설이 존재한다.
- 연인 관계 종료 시 데이터 처리 방식이 불명확하면 부정적 경험(전 연인의 데이터가 그대로 남음)이 생겨 나쁜 입소문이 날 수 있다.
- **위험 수준:** 중간

## 4. 운영 리스크

### 4-1. 팀 역량 및 인력 리스크

- 1인 개발 체제에서 iOS 개발, 백엔드 설계, UI/UX 디자인, 법무 검토, 마케팅을 동시에 감당해야 한다.
- 개발자가 부상, 질병, 번아웃 등으로 개발이 중단되면 서비스 전체가 멈추는 단일 장애점(Single Point of Failure)이 된다.
- 빠른 출시 목표로 인해 기술 부채가 누적되고, 이후 기능 추가나 버그 수정 속도가 크게 저하될 수 있다.
- **위험 수준:** 높음

### 4-2. 자금 조달 리스크

- 수익화 모델 미정 상태에서의 개발·운영 비용(Apple 개발자 계정 연 $99, 서버/BaaS 비용, 앱 마케팅 비용 등)은 자비 부담이 된다.
- 서비스가 성장할수록 Firebase 또는 클라우드 서버 비용이 증가하는데, 수익 없이 비용만 증가하는 구조가 될 수 있다.
- **위험 수준:** 높음

### 4-3. 파트너사 의존도

- Google, Apple, Firebase 세 곳의 정책 변화에 모두 노출되어 있다.
- 각 플랫폼의 정책 변경이나 계정 정지(정책 위반 시) 등으로 서비스가 즉시 영향을 받는다.
- **위험 수준:** 중간

## 6. 대응 전략

### 상위 5개 핵심 리스크 대응 방안

---

#### R01. 보안 취약점 (데이터 무단 접근, OAuth 토큰 탈취)

**대응 방안**
- Firestore/Realtime DB 보안 규칙을 그룹 멤버십 기반으로 엄격하게 설정하고, 출시 전 보안 규칙 유닛 테스트를 필수로 수행한다.
- OAuth 액세스 토큰은 iOS Keychain에만 저장하고, 서버 측에서는 리프레시 토큰을 암호화하여 보관한다.
- 최소 권한 원칙(Least Privilege)을 적용하여 Google Calendar API 스코프는 실제 필요한 범위(calendar.events만 접근 등)로 한정한다.
- 오픈 소스 보안 감사 도구(예: Firebase Rules 테스터, OWASP 체크리스트)를 활용하여 MVP 출시 전 보안 점검을 완료한다.

---

#### R02. 수익화 모델 미정으로 인한 지속 불가 (R03)

**대응 방안**
- MVP 출시 전에 수익화 모델을 확정한다. 권장 모델: **Freemium + 구독제**
  - 무료 플랜: 1개 그룹, 기본 공유 캘린더
  - 프리미엄 플랜 (월 2,900원~4,900원): 다중 캘린더 연동, 일정 히스토리, 테마 등
- Apple IAP 구독 수수료(1년 이후 15%)를 고려한 단가 설정을 사전에 계산한다.
- 출시 후 3개월 내 유료 전환율 목표치(예: 가입 대비 3%)를 설정하고 미달 시 모델 변경을 검토한다.

---

#### R03. 1인 개발 역량 한계 및 단일 장애점 (R04)

**대응 방안**
- 핵심 기능을 3단계로 나누어 단계적으로 출시한다.
  - 1단계: 실시간 공유 캘린더 (핵심 기능만)
  - 2단계: 공동 일정 초대/수락 플로우
  - 3단계: 외부 캘린더 연동 (Google Calendar)
- 코드 기반을 GitHub에 관리하고, 주요 기술 결정 사항을 문서화하여 외주 개발자나 공동 창업자 합류 시 빠르게 온보딩할 수 있도록 한다.
- 번아웃 방지를 위해 주간 개발 시간 한도를 설정하고, 기능 범위를 엄격히 제한한다(스코프 크리프 방지).

---

#### R04. Google API 약관 위반 / OAuth 앱 검토 거부 (R02)

**대응 방안**
- Google Calendar API 연동 기능 추가 전, Google Cloud Console에서 OAuth 앱 검토(OAuth App Verification)를 신청하고 승인을 먼저 받는다. 검토에는 수 주가 소요될 수 있으므로 일정에 반영한다.
- 사용자 데이터 사용 목적을 Google의 "제한된 사용 정책(Restricted Use Policy)"에 맞게 명시하고, 외부 캘린더 데이터를 앱 기능 외 목적으로 활용하지 않는다.
- 외부 캘린더 연동은 MVP 1.0에서 제외하고, OAuth 검토 승인 후 1.1 업데이트로 추가하는 방안을 검토한다.

---

#### R05. 대형 플랫폼의 기능 복제 (R05)

**대응 방안**
- 플랫폼 기능으로 복제하기 어려운 커플/가족 특화 UX(예: 감성적인 일정 표현, 기념일 자동 알림, 공동 일정 수락 인터페이스)에 집중하여 차별점을 강화한다.
- 초기에 충성 사용자 커뮤니티(예: 커플 앱 리뷰 커뮤니티, 육아 커뮤니티)를 형성하여 플랫폼 전환 비용을 높인다.
- 특정 플랫폼(Apple, Google)의 기본 캘린더 기능이 강화될 경우를 대비해, 해당 플랫폼의 API를 적극 활용하여 "플랫폼 위의 레이어" 전략으로 공존 가능성을 확보한다.
