## 1. 테스트 전략

### 1-1. 테스트 범위 및 목표

**대상 제품**: WeCal (함께 쓰는 캘린더) v1.0
**플랫폼**: iOS 16.0 이상, iPhone 전용

**테스트 목표**
- v1.0 출시 범위 핵심 기능(F01~F07) 전체에 대한 품질 검증
- 주요 엣지케이스 및 비기능 요구사항(성능·보안·접근성) 충족 여부 확인
- 앱스토어 심사 통과에 필요한 필수 요건 검증
- 크래시율 0.5% 미만 기준 달성 확인

**테스트 범위 (In Scope)**

| 영역 | 세부 범위 |
|------|-----------|
| 기능 | F01 Sign in with Apple, F02 그룹 생성/초대, F03 캘린더 CRUD, F04 실시간 동기화, F05 공동 일정 초대, F06 푸시 알림, F07 개인정보/약관 |
| 비기능 | 앱 초기 로딩 성능, 동기화 지연, 일정 저장 응답, 크래시율 |
| 보안 | Keychain 토큰 저장, Firestore 보안 규칙, 권한 없는 데이터 접근 차단 |
| 접근성 | VoiceOver, Dynamic Type, 색상 대비, 최소 터치 영역 |
| 오프라인 | 로컬 캐시 기반 조회, 온라인 복귀 시 자동 동기화 |

**테스트 범위 (Out of Scope)**
- F08 구글 캘린더 연동 (v1.1 이연)
- iPad 지원
- 안드로이드 플랫폼
- 유료 구독 결제 흐름 (v1.1 이연)

## 2. 기능 테스트 시나리오

### F01. Sign in with Apple 인증

---

```
TC-001: Sign in with Apple 최초 로그인 및 계정 생성
기능: F01 회원가입 및 로그인
유저 스토리: US-01

사전 조건:
- 앱이 최초 설치된 상태
- 기기에 Apple ID가 설정되어 있음
- 네트워크 연결 정상

테스트 단계:
1. 앱을 실행한다
2. 온보딩 화면에서 "Apple로 로그인" 버튼을 탭한다
3. iOS 시스템 Apple ID 인증 다이얼로그가 표시되면 Face ID 또는 Touch ID로 인증한다
4. 이름 공유 여부를 선택하고 "계속"을 탭한다
5. 인증 완료 후 앱의 다음 화면 진입을 확인한다

기대 결과:
- Apple ID 인증 다이얼로그가 시스템 UI로 정상 표시된다
- 인증 성공 후 신규 사용자 계정이 Firebase Auth에 생성된다
- 이용약관 동의 화면 또는 그룹 설정 온보딩 화면으로 이동한다
- JWT 세션 토큰이 iOS Keychain에 저장된다
우선순위: Critical
```

---

```
TC-002: 기존 사용자 자동 로그인
기능: F01 회원가입 및 로그인
유저 스토리: US-02

사전 조건:
- 이전에 Sign in with Apple로 로그인한 이력이 있는 상태
- 앱이 완전히 종료된 상태 (백그라운드 아님)

테스트 단계:
1. 앱 아이콘을 탭하여 앱을 실행한다
2. 스플래시 화면 이후 표시되는 화면을 확인한다

기대 결과:
- 별도의 로그인 화면이 표시되지 않는다
- 캘린더 메인 화면으로 자동으로 진입한다
- 진입까지 소요 시간이 3초 이내다
우선순위: Critical
```

---

```
TC-003: 기기 교체 후 동일 Apple ID로 로그인 시 계정 복원
기능: F01 회원가입 및 로그인
유저 스토리: US-03

사전 조건:
- 기존에 사용하던 계정 데이터(일정, 그룹)가 Firebase에 저장된 상태
- 새 기기에 앱을 새로 설치한 상태

테스트 단계:
1. 새 기기에서 앱을 실행한다
2. "Apple로 로그인" 버튼을 탭한다
3. 기존 계정과 동일한 Apple ID로 인증한다
4. 메인 화면 진입 후 기존 데이터를 확인한다

기대 결과:
- 기존 그룹 연결 상태가 복원된다
- 이전에 생성한 일정이 캘린더에 정상 표시된다
- 새 계정 생성이 아닌 기존 계정으로 로그인이 처리된다
우선순위: Critical
```

```
TC-004: Apple ID 인증 실패 처리
기능: F01 회원가입 및 로그인
유저 스토리: US-01

사전 조건:
- 앱이 최초 실행된 상태
- 사용자가 Apple ID 인증 다이얼로그에서 "취소"를 선택하는 시나리오

기대 결과:

- 에러 메시지 또는 재시도 안내가 표시된다

우선순위: Critical
```

```
TC-005: 사용자가 이름 제공 거부 시 기본 닉네임 부여
기능: F01 회원가입 및 로그인
유저 스토리: US-01

사전 조건:
- 신규 사용자 최초 로그인 시나리오

1. "Apple로 로그인" 버튼을 탭한다

3. 계정 생성 후 프로필에서 표시 이름을 확인한다

기대 결과:
- "사용자_숫자" 형태의 기본 닉네임이 자동으로 부여된다
- 앱이 크래시 없이 정상적으로 온보딩 화면으로 진행된다
우선순위: High
```

### F02. 그룹 생성 및 초대

---

```
TC-006: 그룹 생성 및 초대 링크 발급
기능: F02 그룹 생성/초대
유저 스토리: US-04

사전 조건:
- 로그인 완료 상태
- 아직 그룹에 참여하지 않은 사용자

테스트 단계:
1. 온보딩 또는 설정 화면에서 "그룹 만들기" 버튼을 탭한다
2. 그룹 이름을 입력하지 않고 "만들기"를 탭한다 (기본값 확인)
3. 생성된 그룹의 초대 링크와 6자리 코드를 확인한다
4. 초대 링크를 복사하여 외부 앱(메시지, 카카오톡 등)으로 공유한다

기대 결과:
- 그룹 이름 미입력 시 "우리의 캘린더"가 기본값으로 설정된다
- 딥링크 형태의 초대 URL과 6자리 숫자/영문 혼합 코드가 생성된다
- 초대 링크가 클립보드에 정상 복사된다
우선순위: Critical
```

---

```
TC-007: 초대 코드 입력으로 그룹 참여
기능: F02 그룹 생성/초대
유저 스토리: US-05

사전 조건:
- 초대를 받을 사용자가 로그인 완료 상태
- 유효한 6자리 초대 코드 보유 (발급 후 72시간 이내)

테스트 단계:
1. 그룹 참여 화면에서 "초대 코드 입력" 탭을 선택한다
2. 6자리 초대 코드를 입력한다
3. "참여하기" 버튼을 탭한다
4. 그룹 캘린더 화면 진입을 확인한다
5. 그룹 생성자의 기기에서 참여 알림 수신을 확인한다

기대 결과:
- 코드 검증 후 그룹 캘린더 화면으로 즉시 이동한다
- 그룹 생성자에게 "{사용자 이름}님이 그룹에 참여했어요" 형태의 푸시 알림이 발송된다
- 신규 참여자가 그룹 멤버 목록에 추가된다
우선순위: Critical
```

---

```
TC-008: 초대 링크 클릭으로 그룹 참여 (앱 설치 상태)
기능: F02 그룹 생성/초대
유저 스토리: US-05

사전 조건:
- 초대 대상 기기에 앱이 설치되어 있고 로그인 완료 상태
- 유효한 초대 딥링크 보유

테스트 단계:
1. 메시지 앱 또는 브라우저에서 초대 링크를 탭한다
2. WeCal 앱이 열리면서 그룹 참여 확인 화면이 표시되는지 확인한다
3. "참여하기"를 탭하여 그룹에 합류한다

기대 결과:
- 딥링크가 앱을 정상적으로 열고 그룹 참여 확인 화면으로 바로 이동한다
- 참여 후 공유 캘린더 화면에 상대방 일정이 표시된다
우선순위: Critical
```

```
TC-009: 5인 초과 그룹 참여 시도 차단
기능: F02 그룹 생성/초대
유저 스토리: US-06

우선순위: Critical
```

### F03. 개인 캘린더 CRUD

---

```
TC-012: 새 일정 생성 (필수 항목만)
기능: F03 개인 캘린더 CRUD
유저 스토리: US-07

사전 조건:
- 로그인 완료 및 캘린더 메인 화면 표시 상태

테스트 단계:
1. 캘린더에서 날짜를 길게 눌러 일정 생성 화면을 연다
2. 제목 필드에 "팀 미팅"을 입력한다
3. 시작 시각을 오후 2:00, 종료 시각을 오후 3:00으로 설정한다
4. 나머지 선택 항목은 그대로 두고 "저장" 버튼을 탭한다
5. 캘린더 화면에서 생성된 일정을 확인한다

기대 결과:
- 저장 버튼 탭 후 1초 이내에 UI에 일정이 반영된다
- 캘린더 해당 날짜·시간대에 "팀 미팅" 일정이 표시된다
- Firestore에 일정 데이터가 정상 저장된다
우선순위: Critical
```

```
TC-014: 종료 시각이 시작 시각보다 이른 일정 저장 시도
기능: F03 개인 캘린더 CRUD
유저 스토리: US-07

사전 조건:
- 일정 생성 또는 수정 화면이 열린 상태

테스트 단계:
```

## 7. 버그 리포트 템플릿

개발팀과의 소통을 위한 표준 버그 리포트 양식입니다. 모든 버그는 이 형식으로 작성하여 이슈 트래커에 등록합니다.

---

```
[버그 리포트]

제목: [기능명] 간략한 버그 설명

버그 ID: BUG-{번호}
발견일: YYYY-MM-DD
보고자: {이름}
심각도: Critical / High / Medium / Low
우선순위: P0 / P1 / P2 / P3
상태: 신규 / 확인 중 / 수정 중 / 수정 완료 / 재현 불가

---

환경 정보
- 기기: (예: iPhone 14 Pro)
- iOS 버전: (예: iOS 17.2.1)
- 앱 버전: (예: v1.0.0 (Build 10))
- 네트워크 환경: (예: Wi-Fi / LTE / 오프라인)
- 계정 상태: (예: 그룹 참여 2인, 일정 50개)

---

재현 단계
1. ...
2. ...
3. ...

---

실제 결과
(실제로 발생한 현상을 구체적으로 기술)

---

기대 결과
(정상적으로 동작해야 하는 결과를 기술)

---

재현율
(예: 5/5 회 재현 / 간헐적 재현)

---

첨부 파일
- 스크린샷 또는 화면 녹화: [파일 첨부]
- 콘솔 로그 / 크래시 리포트: [파일 첨부]
- Charles Proxy 캡처: [해당 시 첨부]

---

추가 메모
(관련 TC 번호, 관련 코드 위치 추정, 임시 우회 방법 등)
```

---

### 심각도 정의

| 심각도 | 정의 | 예시 |
|--------|------|------|
| Critical | 앱 크래시, 데이터 손실, 핵심 기능 완전 불가 | 로그인 불가, 일정 저장 불가, 그룹 참여 불가 |
| High | 주요 기능 부분 동작 불가, 사용자 경험 심각 훼손 | 푸시 알림 미발송, 동기화 지연 5초 초과, 특정 뷰 렌더링 오류 |
| Medium | 기능은 동작하나 사용자 불편, UI/UX 오류 | 경고 메시지 미표시, 잘못된 날짜 표시, 레이아웃 깨짐 |
| Low | 사소한 UI 오류, 오탈자, 미관상 문제 | 텍스트 오탈자, 아이콘 크기 불일치, 색상 미세 차이 |

## 8. 출시 전 QA 게이팅 기준
