## 1. MVP 전략 방향

### 검증하려는 핵심 가설

| 번호 | 가설 | 검증 방법 |
|------|------|-----------|
| H1 | 소규모 그룹(커플/가족)은 전용 공유 캘린더 앱이 있다면 기존 메신저/메모 방식 대신 적극 사용한다 | 7일 이내 재방문율, 일정 생성 횟수 |
| H2 | 실시간 동기화(내가 추가 → 상대방 즉시 반영)가 핵심 사용 동기가 된다 | 앱 내 일정 생성 후 상대방 확인까지 소요 시간, 알림 수신율 |
| H3 | 공동 일정 초대의 수락/거절 플로우가 커플/가족 특화 UX로 기존 앱보다 자연스럽다고 느낀다 | 초대 수락율, 사용자 인터뷰 NPS |

### 타겟 초기 사용자 (Early Adopter)

- **1순위**: 20~30대 커플 (동거 여부 무관), 바쁜 일상 속 데이트/약속 조율에 스트레스를 느끼는 유저
- **2순위**: 맞벌이 부부 (어린 자녀 없는), 가사/외출 일정 공유가 잦은 유저
- **공통 특성**:
  - iOS(아이폰) 사용자
  - 카카오톡 일정 공유나 구두 약속이 반복적으로 누락되는 경험 보유
  - 앱 온보딩에 5분 이상 투자할 의향 없음 (마찰 민감도 높음)
  - Cozi 유료화 이후 대안을 탐색 중인 전환 사용자

### MVP 성공 기준 (정량 지표)

| 지표 | 목표값 | 측정 기간 |
|------|--------|-----------|
| 그룹 연결 완료율 | 앱 설치 사용자의 60% 이상이 파트너와 그룹 연결 완료 | 설치 후 48시간 이내 |
| 7일 리텐션 | 40% 이상 | 출시 후 첫 4주 |
| 1인당 주간 일정 생성 수 | 평균 3건 이상 | 출시 후 4주 평균 |
| 공동 초대 수락율 | 초대 발송 대비 70% 이상 수락 | 출시 후 4주 누적 |
| 앱스토어 평점 | 4.0 이상 | 출시 후 100리뷰 기준 |

## 2. 기능 분류 (MoSCoW)

### Must Have — MVP 필수 기능

핵심 가치 제안을 전달하기 위해 반드시 포함되어야 하는 기능. 없으면 제품이 동작하지 않음.

| 기능 | 설명 | 이유 |
|------|------|------|
| **회원가입 / 로그인** | Sign in with Apple 필수, 이메일 가입 선택 | iOS 앱스토어 정책 필수 요건, 법적 준수 |
| **그룹 생성 및 초대** | 링크/코드 방식으로 파트너 초대, 소규모 그룹(최대 5인) 생성 | 핵심 사용 시나리오의 출발점 |
| **개인 캘린더 CRUD** | 일정 생성/조회/수정/삭제, 하루/주/월 뷰 지원 | 캘린더 앱의 기본 동작 |
| **실시간 공유 동기화** | 그룹 내 일정 변경 즉시 모든 멤버에 반영 (WebSocket 또는 Firebase Realtime) | 핵심 차별점, H1·H2 가설 검증 대상 |
| **공동 일정 초대 + 수락/거절** | 일정에 파트너 초대 → 푸시 알림 발송 → 수락/거절 응답 → 캘린더 반영 | 핵심 차별점, H3 가설 검증 대상 |
| **푸시 알림** | 공동 일정 초대, 일정 당일/전날 리마인더 | 실시간 공유의 체감 품질을 좌우 |
| **개인정보처리방침 / 이용약관** | 앱 내 링크 및 웹페이지 제공 | 법적 필수 요건 |

### Should Have — MVP 이후 빠르게 추가 (v1.1 목표)

사용자 경험을 크게 향상시키거나 초기 이탈을 막기 위해 출시 직후 빠르게 추가해야 할 기능.

| 기능 | 설명 | 우선 추가 이유 |
|------|------|---------------|
| **외부 캘린더 연동 (Google Calendar)** | Google OAuth 앱 검토 승인 후 읽기/쓰기 연동 | 기존 일정 데이터 이전 마찰 제거, 리텐션에 직결 / 리스크 분석상 OAuth 검토 선행 필수 |
| **일정 반복 설정** | 매주/매월 반복 일정 지원 | 루틴 공유(퇴근 시간, 정기 데이트 등) 지원 |
| **일정 색상 태그** | 유저별 또는 카테고리별 색상 구분 | 여러 멤버 일정 혼재 시 가독성 |
| **알림 설정 커스터마이징** | 리마인더 시간 조정 (30분 전, 1시간 전 등) | 알림 피로도 방지 |
| **Apple Calendar 읽기 연동** | 로컬 캘린더 이벤트 가져오기 | iOS 사용자 온보딩 마찰 최소화 |

### Nice to Have — 향후 로드맵 (v2.0+)

차별화를 강화하거나 특정 세그먼트를 위한 부가 기능.

| 기능 | 설명 | 대상 세그먼트 |
|------|------|--------------|
| **D-Day 카운터** | 기념일/이벤트까지 남은 일수 표시 | 커플 |
| **공유 할 일(To-Do) 목록** | 캘린더와 연동된 공동 체크리스트 | 가족, 맞벌이 부부 |
| **기념일 자동 알림** | 생일, 결혼기념일 등 등록 후 매년 자동 리마인더 | 커플, 가족 전체 |
| **일정 메모/사진 첨부** | 일정에 장소 메모, 사진 추가 | 여행, 약속 계획 |
| **위젯 (홈 화면)** | 오늘/내일 공유 일정 위젯 | 헤비 사용자 |
| **프리미엄 구독 결제** | 월 3,900~5,900원 인앱 결제, 추가 기능 제공 | 수익화 전환 대상 |
| **Outlook / iCloud 연동** | 추가 외부 캘린더 지원 | 직장인, Apple 생태계 사용자 |
| **다국어 지원 (영어)** | 글로벌 확장 대비 | 해외 사용자 |

### Won't Have — 이번 MVP에서 명시적으로 제외

| 기능 | 제외 이유 |
|------|-----------|
| **채팅/메시지 기능** | 카카오톡과 중복, 개발 범위 과다. 일정 특화 앱 정체성 유지 |
| **그룹 5인 초과 지원** | 대형 그룹은 기업용 캘린더(Google Workspace 등)와 경쟁 구도. 커플/소가족 집중 |
| **웹 버전 (Web App)** | iOS 우선 전략, 1인 개발 리소스 분산 방지 |
| **Android 버전** | iOS 우선 전략, MVP 검증 후 확장 |
| **AI 일정 추천** | 데이터 부족 및 개발 복잡도 과다, 후순위 |
| **소셜 로그인 (카카오/네이버)** | Sign in with Apple 우선, OAuth 관리 복잡도 증가 방지 |
| **외부 캘린더 연동 (MVP 단계)** | Google OAuth 앱 검토 미승인 리스크, v1.1로 이연 |

## 3. MVP 개발 우선순위

### 개발 순서 및 의존 관계

```
[1단계] 기반 인프라 (병렬 진행 가능)
  ├── 1-A. 백엔드 기초 설정 (Firebase / Supabase 선택 및 프로젝트 설정)
  ├── 1-B. iOS 프로젝트 셋업 (SwiftUI, 패키지 매니저)
  └── 1-C. Sign in with Apple 구현

[2단계] 인증 및 사용자 관리
  └── 2-A. 회원가입/로그인 플로우 완성 → 1-C 완료 후

[3단계] 핵심 캘린더 기능
  ├── 3-A. 개인 캘린더 CRUD + UI (하루/주/월 뷰) → 2-A 완료 후
  └── 3-B. 그룹 생성 및 초대 링크/코드 → 2-A 완료 후

[4단계] 실시간 공유 동기화
  └── 4-A. 실시간 동기화 (Firebase Realtime DB 또는 WebSocket) → 3-A, 3-B 완료 후

[5단계] 공동 초대 플로우
  ├── 5-A. 공동 일정 초대 + 수락/거절 로직 → 4-A 완료 후
  └── 5-B. 푸시 알림 (APNs 연동, 초대/리마인더) → 4-A 완료 후

[6단계] 법적 요건 및 QA
  ├── 6-A. 개인정보처리방침 / 이용약관 페이지
  ├── 6-B. 통합 테스트 및 버그 수정
  └── 6-C. 앱스토어 심사 제출
```

### 단계별 개발 난이도

| 단계 | 주요 작업 | 난이도 | 핵심 리스크 |
|------|-----------|--------|------------|
| 1단계 | 인프라 셋업 | 하 | 서비스 선택 결정 필요 |
| 2단계 | Sign in with Apple | 중 | Apple 정책 준수 필수 |
| 3단계 | 캘린더 CRUD + UI | 중 | 날짜 처리, 타임존 엣지케이스 |
| 4단계 | 실시간 동기화 | 상 | 동시성 충돌, 오프라인 처리, 배터리 소모 |
| 5단계 | 공동 초대 + 푸시 | 중 | APNs 설정, 상태 관리 복잡도 |
| 6단계 | 법적 요건 + 심사 | 하~중 | 앱스토어 심사 반려 가능성 |

## 4. MVP 제약 조건

### 최소 필요 기술 스택

| 영역 | 권장 선택 | 이유 |
|------|-----------|------|
| **iOS 클라이언트** | SwiftUI + Swift | 1인 개발 효율, iOS 우선 전략 |
| **백엔드 / 실시간 DB** | Firebase (Firestore + Realtime Database + FCM) | 실시간 동기화 + 푸시 알림 + 인증을 하나의 플랫폼에서 처리, 1인 개발에 최적 |
| **인증** | Firebase Authentication + Sign in with Apple | 앱스토어 정책 필수, 관리 단순화 |
| **푸시 알림** | Firebase Cloud Messaging (FCM) + APNs | iOS 표준 |
| **상태 관리** | SwiftUI + Combine 또는 TCA (The Composable Architecture) | 실시간 상태 변화 처리 용이 |
| **법적 문서 호스팅** | GitHub Pages 또는 Notion 공개 페이지 | 최소 비용, 빠른 구축 |

> **대안 고려**: Firebase 대신 Supabase(PostgreSQL 기반) 선택 시 실시간 구독(Realtime), Row Level Security로 보안 강화 가능. 단, iOS SDK 성숙도는 Firebase 대비 낮음.

### 예상 개발 기간 (러프한 추정, 1인 개발 기준)

| 단계 | 예상 소요 기간 |
|------|--------------|
| 1단계: 인프라 셋업 | 1주 |
| 2단계: 인증 | 1~2주 |
| 3단계: 캘린더 CRUD + UI | 3~4주 |
| 4단계: 실시간 동기화 | 2~3주 |
| 5단계: 공동 초대 + 푸시 | 2주 |
| 6단계: 법적 요건 + QA + 심사 | 1~2주 |
| **총 예상 기간** | **10~14주 (약 3개월)** |

> 1인 개발, 풀타임 기준. 파트타임 병행 시 5~6개월로 연장 예상.
> 앱스토어 심사 기간(평균 1~3일, 경우에 따라 1주 이상)은 별도 고려 필요.

## 5. 검증 계획

### MVP 검증 방식

#### Phase 0 — 출시 전 (클로즈드 베타)
- **대상**: 지인 커플/부부 10~20쌍 (40명 내외) TestFlight 배포
- **기간**: 2주
- **수집 데이터**:
  - 그룹 연결 완료율
  - 일정 생성 빈도
  - 공동 초대 수락율
  - 앱 크래시 리포트 (Firebase Crashlytics)
- **정성 피드백**: 1:1 인터뷰 또는 설문 (온보딩 경험, 실시간 동기화 체감 속도)

#### Phase 1 — 앱스토어 출시 직후 (오픈 베타)
- **마케팅 채널**: 커플 커뮤니티(결혼/연애 카페), SNS(인스타그램 릴스, 틱톡) 자연 바이럴
- **핵심 지표 모니터링** (Firebase Analytics + Mixpanel 무료 플랜):
  - 온보딩 퍼널: 앱 설치 → 회원가입 → 그룹 생성 → 파트너 연결 → 첫 일정 생성
  - 7일 / 30일 리텐션
  - 일정 생성 빈도 (DAU 대비 이벤트 수)
  - 공동 초대 발송 수 및 수락율

#### 피봇 기준점

| 상황 | 판단 기준 | 대응 방향 |
|------|-----------|-----------|
| 그룹 연결 완료율 40% 미만 | 온보딩 마찰 과다 | 초대 플로우 대폭 단순화, 링크 공유 방식 개선 |
| 7일 리텐션 20% 미만 | 핵심 가치 미전달 | 사용자 인터뷰 집중 실시, 실시간 동기화 안정성 점검 |
| 공동 초대 수락율 50% 미만 | 초대 UX 또는 알림 문제 | 알림 메시지 A/B 테스트, 수락 UI 개선 |
| 일정 생성 주 1건 미만 | 앱 사용 동기 부재 | 타겟 세그먼트 재검토, 가치 제안 메시지 수정 |
| 실시간 동기화 지연 빈발 | 기술 구현 이슈 | Firebase 플랜 업그레이드 또는 아키텍처 재검토 |

### 검증 이후 다음 단계 결정 기준

- **Go (v1.1 진행)**: 7일 리텐션 40% 이상, 그룹 연결 완료율 60% 이상 동시 달성
- **Pivot**: 위 지표 중 하나 이상 피봇 기준 하회 시, 2주 인터뷰 집중 후 방향 재정의
- **Stop**: 출시 후 8주 내 활성 그룹 50쌍 미만 시, 타겟 재정의 또는 프로젝트 재검토

## 요약: MVP 핵심 원칙

1. **범위를 좁게** — 외부 캘린더 연동(Google OAuth)은 v1.1로 이연하여 리스크 제거 및 개발 기간 단축
2. **실시간 동기화가 전부** — 이 기능이 불안정하면 제품 존재 이유 없음. 품질에 최우선 집중
3. **온보딩 5분 이내** — 파트너 연결까지의 마찰을 최소화하지 않으면 이탈률이 즉각 상승
4. **법적 요건 선제 준비** — Sign in with Apple, 개인정보처리방침은 앱스토어 심사 필수 요건으로 개발 초기부터 포함
5. **측정 가능한 지표 설정** — 성공/피봇 기준을 사전에 명시하여 감정적 판단 배제
