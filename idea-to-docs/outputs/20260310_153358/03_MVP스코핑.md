# MVP 스코핑 보고서
생성일: 2026-03-10

---

## 제품 개요

**Baskit** — Siri 음성 입력, 공유 리스트, 쿠팡 파트너스 기반 상품 링크 추천을 결합한 스마트 장보기 앱. 1인 개발 · 저비용 인프라 · iOS 우선 출시 전략.

## 1. MVP 전략 방향

### 1-1. 검증하려는 핵심 가설

| # | 가설 | 검증 방법 | 성공 기준 |
|---|------|-----------|-----------|
| H1 | 사용자는 Siri 음성 입력으로 아이템을 추가하는 습관을 형성한다 | 전체 아이템 추가 경로 중 Siri 비율 측정 | Siri 경유 추가 비율 30% 이상 |
| H2 | 쿠팡 파트너스 상품 링크를 클릭하고 실제 구매로 이어진다 | 링크 CTR 및 쿠팡 파트너스 전환율 실측 | CTR 25% 이상, CVR 15% 이상 |
| H3 | 공유 리스트가 커플/부부 사이에서 바이럴 성장을 유도한다 | 초대 링크 경유 신규 가입 비율 추적 | 신규 가입자 중 30% 이상이 초대 링크 경유 |
| H4 | 핵심 루프(기억 → 리스트 → 추천 → 구매)가 주 1회 이상 반복 사용된다 | D7 / D30 리텐션 추적 | D7 리텐션 50%, D30 리텐션 30% |

### 1-2. 타겟 초기 사용자 (Early Adopter)

1순위 — **함께 장보는 커플/부부**
- 공유 리스트의 직접적인 수혜자
- 한 명만 설치하면 파트너 초대로 바이럴 발생
- 주 1~2회 정기 장보기 패턴 → 높은 DAU 기대

2순위 — **iOS 헤비 유저이자 Siri 습관이 있는 1인 가구 20~30대**
- SiriKit 연동의 차별점을 가장 빠르게 인지하는 세그먼트
- 앱스토어 리뷰 작성 및 SNS 공유 활동성 높음

3순위 제외 — "여러 쇼핑몰 비교형 쇼핑 효율파"는 MVP에서 유료 구독 기능이 없으므로 2차 업데이트 이후 타겟으로 이동.

## 2. 기능 분류 (MoSCoW)

### Must Have — MVP 필수 기능

> 없으면 핵심 가치 제안이 전달되지 않거나 앱 자체가 동작하지 않는 기능

#### M1. 장보기 리스트 CRUD

- 아이템 추가 / 수정 / 삭제 / 체크(구매 완료 표시)
- 카테고리 자동 분류 (식품, 생활용품, 기타) — 규칙 기반 키워드 매핑
- 이유: 장보기 앱의 존재 이유. 이것 없이는 어떤 기능도 의미 없음.

#### M2. Siri 음성 입력 연동 (App Intents, iOS 16+)

- "Baskit에 두부 추가해줘" 형태의 자연어 명령으로 아이템 추가
- App Intents 프레임워크 기반 구현 (SiriKit Lists 도메인 대신)
- 이유: Baskit의 가장 강력한 차별점. 경쟁 앱 대비 유일한 국내 Siri 한국어 최적화 장보기 앱 포지셔닝의 핵심.

#### M3. 공유 리스트 (2인 실시간 동기화)

- 초대 링크 생성 → 파트너가 링크 클릭 후 앱 설치 및 리스트 공유
- 아이템 추가/체크 변경 사항 실시간 반영
- 이유: 커플/부부 타겟의 핵심 페인포인트 해결. 공유 기능이 없으면 1인 리스트 앱과 차별화 불가. 초대 구조가 바이럴 성장의 유일한 채널.

#### M4. 쿠팡 파트너스 상품 링크 추천

- 아이템 입력 시 쿠팡 파트너스 API 기반 상품 검색 결과 1~3개 표시
- 상품명, 가격, 이미지 썸네일, 링크 표시
- 링크 탭 시 Safari로 쿠팡 상품 페이지 이동
- 이유: 수익 모델의 핵심. MVP의 유일한 수익원(어필리에이트). 없으면 비즈니스가 성립하지 않음.

#### M5. Apple 계정 기반 간편 로그인 (Sign in with Apple)

- 별도 회원가입 없이 Apple ID로 즉시 시작
- 공유 리스트 연동을 위한 최소한의 사용자 식별 필요
- 이유: 앱스토어 가이드라인상 소셜 로그인을 제공하는 경우 Sign in with Apple 필수. 별도 이메일/비번 가입 UX는 온보딩 이탈률을 높임.

#### M6. 기본 구매 이력 저장

- 체크 완료된 아이템을 날짜와 함께 이력으로 저장
- 이력 기반 "자주 사는 것" 목록 표시 (단순 빈도 카운트)
- 이유: 추후 추천 고도화의 데이터 기반. MVP에서는 단순 빈도 기반으로 구현하여 AI 비용 없이 개인화 흉내 가능.

### Should Have — 2차 업데이트 (출시 후 1~3개월)

> 사용자 경험을 크게 향상시키거나, 유료 구독 전환을 위한 핵심 유인

#### S1. 네이버 쇼핑 API 연동 (가격 비교 미리보기)

- Naver Open API 쇼핑 검색으로 쿠팡 외 가격 비교 정보 노출
- 유료 구독 전환 유인 ("더 많은 쇼핑몰을 비교하려면 구독하세요") 게이트웨이 역할
- 우선순위 이유: 유료 구독 가치 명제를 사용자에게 처음으로 보여주는 기능. 구독 CTA 없이는 수익 다각화 불가.

#### S2. 유료 구독 결제 (인앱 구매, 월 3,900원)

- App Store 인앱 구매로 구독 처리 (애플 결제 시스템 사용)
- 구독 시 멀티 쇼핑몰 가격 비교 기능 잠금 해제
- 우선순위 이유: 장기 수익 모델의 실제 검증. 어필리에이트만으로는 수익 안정성 부족.

#### S3. 위젯 지원 (iOS 홈 화면 / 잠금 화면 위젯)

- 홈 화면에서 리스트 아이템 빠른 확인 및 체크
- 이유: 장보기 중 앱 진입 마찰 감소. D7 리텐션 향상에 기여하는 핵심 UX 요소.

#### S4. 재구매 알림 (푸시 알림)

- 이전 구매 이력 기반으로 "지난주에 구매했던 달걀이 떨어질 때가 됐어요" 형태의 알림
- 규칙 기반 주기 계산 (평균 구매 간격 + 3일)
- 이유: 리텐션 향상의 핵심 수단. 사용자가 앱을 능동적으로 열지 않아도 다시 유입 가능.

#### S5. 리스트 템플릿 (자주 사는 묶음 저장)

- "주간 기본 장보기" 같은 아이템 묶음을 저장하고 한 번에 추가
- 이유: 반복 장보기 패턴을 가진 사용자의 가장 강한 니즈. 리텐션 및 Siri 활용 시나리오 확장.

### Nice to Have — 장기 로드맵 (출시 후 6개월 이상)

> 차별화를 강화하거나 특정 세그먼트를 위한 부가 기능

#### N1. AI 기반 상품 추천 고도화

- LLM(GPT-4o mini 또는 Claude Haiku)을 활용한 자연어 기반 상품 매칭
- 예: "두부찌개 끓이려는데 뭐 사야 해?" → 재료 리스트 자동 생성
- 장기 이유: LLM 비용 리스크(R4)가 MVP 단계에서는 수익 대비 부담. 유료 구독자 전용 기능으로 도입.

#### N2. 마켓컬리 / SSG / 지마켓 연동 (유료 구독 전용)

- 공식 API 파트너십 협의 후 순차 연동
- 장기 이유: 공개 API 없음(마켓컬리, 다이소). 파트너십 협의에 최소 3~6개월 소요 예상.

#### N3. 레시피 연동

- 레시피 앱 또는 유튜브 요리 영상과 연동하여 재료 리스트 자동 생성
- 장기 이유: 타사 플랫폼 API 의존성 추가. 핵심 루프 검증 후 부가 가치 확장 순서.

#### N4. 그룹 공유 (3인 이상)

- 가족 단위, 룸메이트 그룹을 위한 리스트 다자간 공유
- 장기 이유: MVP는 커플/2인 공유로 충분. 다자간 실시간 동기화는 아키텍처 복잡도 증가.

#### N5. 안드로이드 앱

- iOS 검증 후 플러터(Flutter) 또는 리액트 네이티브로 크로스 플랫폼 확장
- 장기 이유: Siri 연동이 iOS에 종속. Android에서는 차별화 포인트 재설계 필요.

#### N6. 정기 배송 자동 연결

- 반복 구매 패턴 감지 → 쿠팡 로켓배송 정기 구독 자동 추천
- 장기 이유: 쿠팡 파트너스 API 이상의 파트너십 수준 필요. 추가 협상 프로세스 요구.

#### N7. 오프라인 바코드 스캔

- 실물 상품 바코드 스캔 → 리스트에 자동 추가
- 장기 이유: 카메라 권한, 바코드 DB 연동 등 추가 개발 비용 발생. 온라인 장보기 타겟 우선.

| 기능 | 제외 이유 |
|------|-----------|
| 다이소몰 연동 | 공개 API 없음. 웹 스크래핑은 이용약관 위반 및 법적 리스크(R1) |
| 마켓컬리 연동 | 동상동 이유. 파트너십 협의 선행 필요 |

| Android 앱 | Siri 연동이 iOS 전용. Android 차별점 재설계 없이 출시 시 품질 저하 |
| 가계부/지출 분석 | 핵심 루프(리스트 → 추천 → 구매)와 직접 연결되지 않는 인접 기능. 집중력 분산 |
| 소셜 피드 / 커뮤니티 | 1인 개발 리소스로 감당 불가. 콘텐츠 모더레이션 운영 부담 |

| 가격 히스토리 추적 | 각 쇼핑몰 가격 데이터 지속 저장 = 저작권 이슈(R11) 및 스토리지 비용 급증 |

## 3. MVP 개발 우선순위

### 개발 순서 및 의존 관계

```
[1단계] 기반 인프라 (1~2주)
  ├── Rails 백엔드 초기 셋업 (Supabase PostgreSQL 연동)
  ├── Sign in with Apple 인증 구현
  └── 기본 API 엔드포인트 (리스트 CRUD)

[2단계] iOS 핵심 기능 (2~4주)
  ├── SwiftUI 기반 리스트 UI 구현 (M1)
  ├── App Intents Siri 연동 (M2)  ← [1단계] 완료 필요
  └── 공유 리스트 실시간 동기화 (M3)  ← [1단계] 완료 필요

[3단계] 수익화 연동 (4~5주)
  └── 쿠팡 파트너스 API 상품 추천 (M4)  ← [2단계] 완료 필요

[4단계] 데이터 기반 (5~6주)
  └── 구매 이력 저장 및 "자주 사는 것" 표시 (M6)  ← [3단계] 완료 필요

[5단계] 출시 준비 (6~7주)
  ├── 앱스토어 심사 제출 준비 (심사 노트 작성, 어필리에이트 구조 명확화)
  ├── 개인정보처리방침 및 이용약관 페이지
  └── TestFlight 베타 테스트 (커플/부부 10팀 내외)
```

### 예상 개발 난이도

| 기능 | 난이도 | 근거 |
|------|:------:|------|
| 리스트 CRUD (M1) | 하 | SwiftUI 기본 패턴, 표준 REST API |
| Sign in with Apple (M5) | 하 | Apple 공식 SDK, 문서화 잘 됨 |
| 구매 이력 저장 (M6) | 하 | 단순 DB 적재, 빈도 카운트 |
| 쿠팡 파트너스 API (M4) | 중 | API 승인 프로세스 + Rate Limit 설계 필요 |
| 공유 리스트 동기화 (M3) | 중 | Supabase Realtime 활용, 동시성 처리 필요 |
| Siri App Intents 연동 (M2) | 상 | iOS 16+ App Intents 프레임워크 학습 비용, 한국어 NLU 튜닝 필요 |

## 4. MVP 제약 조건

### 기술 스택 추천 (1인 개발 · 저비용 기준)

| 레이어 | 선택 기술 | 이유 |
|--------|-----------|------|
| iOS 앱 | Swift / SwiftUI | Siri App Intents는 SwiftUI 네이티브 구현 필수. React Native 등 크로스 플랫폼으로는 SiriKit 완전 구현 어려움 |
| 백엔드 | Ruby on Rails (API 모드) | 제품 브리프의 기술 스택. 빠른 프로토타이핑, Gem 생태계 풍부 |
| 데이터베이스 | Supabase (PostgreSQL) | 무료 플랜 50K MAU, Realtime 기능 내장으로 공유 리스트 동기화 구현 용이. 1인 개발 운영 부담 최소 |
| 인증 | Supabase Auth + Sign in with Apple | 별도 인증 서버 불필요, Apple 가이드라인 준수 |
| 실시간 동기화 | Supabase Realtime | 공유 리스트 변경 사항 실시간 반영. Firebase 대비 PostgreSQL 직접 연동으로 데이터 정합성 높음 |
| 호스팅 | Railway 또는 Fly.io | Rails 앱 배포 용이, 소규모 트래픽 무료/저비용 플랜 존재 |
| 상품 추천 | 쿠팡 파트너스 API (규칙 기반) | LLM 비용 없음. 키워드 기반 API 호출로 MVP 충분 |
| 모니터링 | Sentry (무료 플랜) | 1인 개발에서 에러 트래킹 필수. 무료 플랜으로 충분 |
| 분석 | Mixpanel 또는 Amplitude (무료 플랜) | 이벤트 기반 리텐션 분석. 핵심 가설 검증용 코호트 분석 |

### 예상 개발 기간 (러프한 추정)

- MVP 총 개발 기간: **6~8주** (1인 풀타임 기준)

## 5. 검증 계획

### 검증 단계 및 방법

#### Phase 0 — 출시 전 베타 (TestFlight, 2주)

- 모집 대상: 커플/부부 10팀 (20명), 1인 가구 10명
- 모집 채널: 개인 SNS, 지인 네트워크
- 측정 항목: 온보딩 완료율, Siri 사용 시도율, 공유 리스트 활성화율, 치명적 버그 식별
- 합격 기준: 온보딩 완료율 80% 이상, 치명적 버그 0건

#### Phase 1 — 출시 후 4주

- 채널: 앱스토어 자연 유입, 커플/부부 커뮤니티 (네이버 카페 "신혼부부", "맘카페"), 1인 가구 라이프스타일 인스타그램 바이럴
- 핵심 측정: 가설 H1~H4 동시 추적
- 의사결정 기준: D7 리텐션 50% 미달 시 온보딩 UX 즉시 개선

#### Phase 2 — 출시 후 4~12주

- Siri 사용률 30% 미달 시: 텍스트 입력 UX 강화 및 Siri 온보딩 가이드 개선
- 쿠팡 링크 CTR 25% 미달 시: 상품 추천 노출 위치 및 문구 A/B 테스트
- 공유 초대 30% 미달 시: 초대 UX 단순화 및 초대 인센티브(예: 추천 상품 추가 제공) 검토

### 피봇 기준점

| 상황 | 피봇 방향 |
|------|-----------|
| Siri 사용률 10% 미만이 2개월 지속 | Siri를 주력 마케팅에서 제외. 텍스트/위젯 UX 중심으로 재포지셔닝 |
| 쿠팡 파트너스 정책 변경으로 어필리에이트 수익 소멸 | 유료 구독 전환 가속화. 무료 티어 기능 축소 후 구독 전환율 향상에 집중 |
| D30 리텐션 15% 미만 3개월 지속 | 핵심 루프 재검토. 재구매 알림(S4) 및 위젯(S3) 앞당겨 개발 |
| 공유 리스트 활성 팀 100팀 미달 (3개월) | 커플/부부 타겟 피봇. 1인 가구 + Siri 중심의 개인용 앱으로 집중 |
| 앱스토어 어필리에이트 구조 심사 리젝 반복 | 외부 링크 구조 제거, 쿠팡 딥링크 방식으로 변경하거나 수익 모델을 구독 전용으로 전환 |

### 기능 완성도
